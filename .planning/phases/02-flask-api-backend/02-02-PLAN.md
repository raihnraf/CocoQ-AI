   - Validate numeric ranges: temperature (90-140), moisture (0-10), pH (2-9), color_score (0-100), cooking_time (30-150), dryness_level (1-5)

   - Call generate_recommendation(grade, input_dict) → get recommendation string
   - Return 200 JSON: `{"grade": grade, "confidence": confidence, "recommendation": recommendation}`
   - On any exception: return 500 JSON `{"error": "Prediction failed: {message}"}`

<task type="auto">
  <name>Task 2: Create BatchAPI and FeatureImportanceAPI endpoints (API-02, API-03, ML-03)</name>
  <files>app/api/routes.py</files>
  <action>Add to app/api/routes.py:

BatchAPI(MethodView) with methods = ['GET', 'POST']:
1. `GET /api/batches` — paginated batch history:
   - Accept query params: `page` (default 1), `per_page` (default 20, max 100)
   - Query SQLite: `SELECT * FROM batches ORDER BY created_at DESC LIMIT ? OFFSET ?`
   - Also query total count: `SELECT COUNT(*) FROM batches`
   - Return 200 JSON: `{"batches": [...], "total": count, "page": page, "per_page": per_page, "pages": ceil(total/per_page)}`
   - Convert sqlite3.Row objects to dicts for JSON serialization
2. `POST /api/batches` — create new batch record:
   - Parse JSON body with `request.get_json()`
   - Validate required fields: temperature, moisture, pH, color_score, cooking_time, supplier_origin, dryness_level, visual_inspection (same validation as predict endpoint)
   - Optional fields: batch_id (auto-generate as 'B-YYYY-NNN' if not provided), production_date (default to today)
   - Insert into batches table using parameterized query: `INSERT INTO batches (batch_id, production_date, temperature, moisture, ph, color_score, cooking_time, supplier_origin, dryness_level, visual_inspection) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)`
   - Return 409 JSON `{"error": "Batch ID already exists"}` if batch_id duplicate
   - Return 201 JSON with the created record including its id and created_at timestamp
   - On validation error: return 400 JSON `{"error": "..."}`

FeatureImportanceAPI(MethodView) with methods = ['GET']:
1. `GET /api/feature-importance`:
   - Load model from flask.current_app.config['MODEL']
   - Call get_feature_importance(pipeline) → list of {feature, importance} dicts
   - Return 200 JSON: `{"features": [{feature, importance}, ...]}`
   - On error: return 500 JSON `{"error": "Failed to load feature importance"}`

Register both with blueprint:
- `api_bp.add_url_rule('/batches', view_func=BatchAPI.as_view('batches'))`
- `api_bp.add_url_rule('/feature-importance', view_func=FeatureImportanceAPI.as_view('feature_importance'))`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API endpoints | All JSON input is untrusted — must validate type, range, and allowed values |
| API → SQLite database | SQL injection risk if user input reaches queries without parameterization |
| API → Model pipeline | Malformed input could cause prediction crashes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Injection | BatchAPI POST | mitigate | All SQL uses parameterized queries with ? placeholders — zero string interpolation for values |
| T-02-06 | Spoofing | PredictAPI POST | mitigate | Strict input validation: type checks (isinstance float/int), range bounds, allowed categorical values — reject before processing |
| T-02-07 | Information Exposure | BatchAPI GET | accept | Batch data is simulated production data, no PII — pagination prevents data dumping |
| T-02-08 | Denial of Service | BatchAPI GET | mitigate | per_page capped at 100, pagination enforced — prevents single-request full table scan |
| T-02-09 | Tampering | BatchAPI POST (batch_id) | mitigate | batch_id format validated (B-YYYY-NNN pattern), auto-generated if not provided — prevents arbitrary ID injection |
</threat_model>
